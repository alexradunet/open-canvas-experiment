ES Modules: The Bundler Replacement

In a framework project, webpack or Vite resolves your imports, bundles files, and manages scope. In a no-build app, the browser does all of this natively. ES modules are the mechanism.

The mental model

A module is a file. That's it. One file, one module. The browser:

  1. Sees <script type="module" src="main.js">
  2. Fetches main.js
  3. Reads its import statements
  4. Fetches those files (and their imports, recursively)
  5. Executes everything in dependency order

No config file. No resolver plugin. No bundle output. The URL is the module ID.

The three rules

1. Modules have their own scope. Top-level const x = 1 in a module is invisible to other modules unless you export it. No IIFEs, no namespace objects, no accidental globals.

// storage/vault-path.js
const SEPARATOR = '/';  // private to this module

export function normalizePath(raw) {
  return raw.split(SEPARATOR).filter(Boolean).join('/');
}

2. Modules execute once. If ten files import vault-path.js, the browser fetches and evaluates it once. Every importer gets the same live bindings. This replaces the "singleton module" pattern bundlers give you.

3. Modules are always strict. No "use strict" needed. Ever. The module system enforces it.

Static vs. dynamic imports

Static — resolved at load time, before your code runs:

import { normalizePath } from './storage/vault-path.js';
import * as codec from './storage/entity-codec.js';

Dynamic — resolved at runtime, returns a Promise:

const { openVault } = await import('./storage/indexeddb-vault.js');

Use static imports for your dependency graph (the browser preloads them). Use dynamic import() for code you only need on demand — a dialog, a heavy feature, a conditional path.

Import paths are URLs

This trips up everyone coming from bundlers. In the browser, import paths are real URLs relative to the importing file. They must include the file extension:

// WRONG (bundler habit)
import { foo } from './utils';
import { bar } from 'lodash';

// RIGHT (browser native)
import { foo } from './utils.js';
import { bar } from '../vendor/lodash-es/lodash.js';

There is no node_modules resolution. There is no extension inference. The path you write is the path the browser fetches.

Balaur connection

Look at main.js in this repo. It's a single <script type="module"> entry point. Every import is a relative path with .js extension. The browser builds the entire dependency graph from that one file. No webpack config exists anywhere.

Retrieval practice

Check your understanding

Q1. Two modules both import ./state.js. How many times does the browser evaluate state.js?

Correct. The module registry caches the evaluated module. All importers share the same live bindings.
Not quite. The browser maintains a module registry — a module is fetched and evaluated exactly once, then cached for all subsequent importers.

Q2. What's wrong with this import in a browser-native module?

import { render } from './components/card';
Correct. The browser needs a full URL path — no extension inference. It must be './components/card.js'.
The browser treats import paths as URLs. Unlike bundlers, it won't guess .js — you must write './components/card.js' explicitly.

Q3. When should you use await import('./x.js') instead of a static import?

Correct. Dynamic import defers fetching until runtime — ideal for lazy-loading features, conditional code paths, or heavy modules not needed at startup.
Dynamic import() returns a Promise resolved at runtime. Use it for code you don't need at page load — dialogs, optional features, heavy computations. Static imports let the browser preload the full graph upfront.
